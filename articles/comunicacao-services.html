Comunicação com Services em Angular

Preview aqui

Por Camila Matias | Desenvolvedora Front End

Publicado: 24 Junho 2024

Navegue pelo conteúdo

  1. Introdução
  2. HttpClient e Async Pipe
  3. Subject e BehaviorSubject
  4. Signals

Os services, em Angular, são uma ferramenta importantíssima para compartilhamento de dados. Em componentes que não possuem conexão hierárquica, principalmente, esta ferramenta auxilia a manter diferentes partes da aplicação com uma única fonte de dados, o que gera um código com maior manutenibilidade e menor propensão a erros.

Uma das grandes utilidades dos services em Angular é a utilização do HttpClient para a realização de requisições HTTP. Dado que o HttpClient foi incluso nos providers do app.config, os métodos foram inicialmente definidos dentro da service, e a service foi injetada no componente, é possível dispor os dados resultantes de um método GET, por exemplo, de duas formas:

  • Com o subscribe, gerando a resposta da requisição, atribuindo-a à uma variável e inserindo essa variável no template com o double moustache ou de qualquer outra forma que se deseje.
  • Com o async pipe, em que não se faz o subscribe diretamente (o pipe irá realizar este trabalho). É necessário apenas que o Observable da requisição seja atribuído à uma variável e esta seja inserida no template com o async pipe. Quando a requisição estiver completa, o conteúdo irá automaticamente ser disposto na tela. Para alguns tipos de resposta, pode ser necessário associar o async pipe com o json pipe para que os dados sejam corretamente exibidos.

Com um mesmo service sendo injetado em diferentes componentes, diferentes partes da aplicação sempre possuirão os mesmos dados, o que previne inconsistências e traz uma experiência mais fluida ao usuário.

Além de dados provenientes de requisições HTTP, os services podem ser veículos para outros tipos de comunicações entre componentes, como as com Subjects e BehaviorSubjects. Os Subjects são um tipo de Observable que podem emitir eventos ou valores para vários observadores de uma vez só (o chamado multicast) usando a função next(), porém apenas após a inscrição. BehaviorSubjects, por sua vez, são tipos de Subject que requerem valores iniciais e em que os "inscritos", ao se inscreverem, recebem automaticamente o último valor repassado pelo BehaviorSubject.

Estes dois artefatos são utilizados para repasse de uma mesma informação (o Subject/BehaviorSubject) para vários componentes (os inscritos), porém o Subject é utilizado quando não há uma preocupação com o estado inicial do dado e o BehaviorSubject, quando há.

Para criar uma comunicação entre componentes via service com Subject ou BehaviorSubject, basta que um dado destes tipos seja inicializado no service e que os componentes a exibirem os dados criados façam o subscribe do Subject ou do BehaviorSubject (dado que o service esteja injetado nos componentes). Com o subscribe feito, quando algum componente editar o dado, este será atualizado também no service, que irá possibilitar o repasse da atualização para quaisquer componentes que tiverem feito o subscribe.

Com a atualização para a versão 17 do Angular e a elevação do status do conceito dos Signals, se tornou importante avaliar se a programação reativa realizada com RxJS (Observables, Subject e BehaviorSubject) não seria melhor realizada com signals. A resposta definitiva depende do caso e do programador, mas já se vêem ganhos na facilidade de escrita de código, principalmente para programadores iniciantes em Angular.

Para criar uma comunicação entre componentes com signals via services, basta que se defina no service o signal inicial. Injetando o service nos componentes a exibirem os dados e inicializando variáveis que apontem para o dado advindo do service, caso um componente utilize o método set() para modificar o signal, esta modificação será repassada por meio do service.

Com a criação de um service com a inicialização dos dados e métodos de alteração e um componente de edição com o service injetado, qualquer componente criado para exibição dos dados que fizer o subscribe nos Subject e BehaviorSubject, além do apontamento para o signal em uma variável, poderá receber quaisquer alterações realizadas pelo componente editor. Este componente editor, por sua vez, terá inputs de texto para coleta do valor pelo usuário, por exemplo, e botões para chamada das funções de alteração.
